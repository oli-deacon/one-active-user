# One Active User

## Product Specification

### Summary

`One Active User` is a curated editorial website for people who build personal software: tools made to reduce friction in their own lives, even if they only ever serve one person.

The site should feel less like a startup showcase and more like a thoughtful home for a specific philosophy of software:

- software for a market of one
- software built in response to lived friction
- software that earns its keep through usefulness, not scale

The first version should validate that this framing resonates, that people understand it quickly, and that it attracts high-quality stories and submissions.

## Brand

### Name

`One Active User`

### Why this name

It is the strongest option because it is:

- distinctive
- memorable
- conceptually rich
- aligned with the central thesis of the project

It also subtly flips familiar software/product language into a more human and personal framing.

### Working tagline options

- A home for personal software
- Software for a market of one
- Tools built to reduce personal friction
- Where personal software lives

Recommended default:

`A home for personal software`

## Product Thesis

Most software spaces reward scale, polish, and startup potential.

`One Active User` should reward:

- specificity
- usefulness
- honesty
- personal fit
- reflective storytelling

This is not primarily a site about AI, coding tools, indie hacking, or launching products.

Those things may be adjacent, but the core subject is personal software: software made because someone encountered recurring friction in their own life and had the means to build the exact thing they needed.

## Goals

### Primary goals for MVP

- Explain the concept in under 10 seconds.
- Publish a small set of strong seed content.
- Showcase personal software examples in a structured way.
- Accept outside submissions through a simple workflow.
- Establish a recognizable editorial lens around personal friction and the one-active-user idea.

### Secondary goals

- Build an audience around the philosophy of personal software.
- Create a repeatable publishing format for future stories and build profiles.
- Give people permission to build software for themselves without needing to justify scale.

## Non-goals

The MVP should not include:

- public user accounts
- comments
- likes or upvotes
- following/follower systems
- community forums
- instant self-publishing
- complex creator profiles
- monetization features
- advanced search/discovery systems

This should begin as a curated editorial product, not a community platform.

## Audience

### Primary audience

- people building small personal tools with modern coding tools
- technical tinkerers who make software for themselves
- designers, product people, and curious builders who do not necessarily identify as software engineers

### Secondary audience

- readers interested in the philosophy of personal software
- people curious about AI-assisted software creation
- people looking for examples, prompts, or encouragement to build their own tools

## Positioning

### Core positioning statement

`One Active User` is a home for personal software: tools, stories, and ideas from people reducing friction in their own lives, one active user at a time.

### Editorial distinction

This should not feel like:

- an AI toy gallery
- an indie hacker launch board
- a prompt directory
- a generic SaaS/product showcase

This should feel like:

- a thoughtful editorial site
- a practical gallery of useful personal tools
- a place where small, specific software is taken seriously

### 1. Homepage

Purpose:

Suggested content:

- what personal software means

### 3. Stories

Purpose:

- publish essays, reflections, and writeups
- establish voice and editorial point of view

Examples:

- The advent of personal software
- What one active user means
- Why software does not need a market to be worth building

### 4. Builds

Purpose:

- showcase personal software examples in a structured, browsable way
- help readers quickly understand the problem, the solution, and the usefulness

Examples:

- landing-brief
- strava-insights
- kit-check

### 5. Submit

Purpose:

- make contributions easy
- keep the submission process lightweight
- maintain curation

The MVP submission flow should likely be form-based and manually reviewed.

### 6. Start Here (optional)

Purpose:

- give new readers a clear onboarding path
- help visitors understand what counts as personal software
- lower the intimidation factor for potential contributors

## Submission Model

The submission format is part of the product. It should reinforce the editorial lens.

### Build submission questions

- What did you build?
- What personal friction were you trying to reduce?
- Why did existing software not solve it well enough?
- Who is it for?
- How are you using it now?
- Is anyone else using it?
- What did building it teach you?
- Do you have a link or screenshots?

### Story submission questions

- What is the piece about?
- Why does it belong on One Active User?
- What experience or example does it draw from?
- Do you have a draft or outline?

### Submission principles

- low-friction
- lightweight
- curated rather than automatic
- no need for polished language at submission stage

## Suggested User Flows

### Flow 1: Reader discovers the concept

- lands on homepage
- understands the idea quickly
- clicks into a featured story or build
- browses more examples
- leaves with a clear sense of the site's philosophy

### Flow 2: Builder wants to submit

- lands on build or manifesto content
- recognizes that something they made fits the concept
- clicks submit
- completes a simple form
- gets clear confirmation that submissions are curated and reviewed

### Flow 3: Curious newcomer wants to build

- reads manifesto or Start Here page
- sees examples of personal software
- understands that narrow and personal is the point
- feels encouraged to build something small for themselves

The MVP should optimize for speed, editorial control, and easy iteration.

Recommended approach:

- content-driven site
- static or mostly static rendering
- curated publishing workflow
- simple form-backed submissions

Suggested stack shape:

- Next.js

## Success Criteria

The MVP is successful if it proves:

- visitors understand the concept quickly
- the framing resonates
- the initial content feels coherent and distinctive
- contributors want to submit tools or stories
- the site feels like a real home for this idea rather than a one-off essay

Suggested early indicators:

- 3-5 seed pieces published
- a working submission flow
- several quality submissions or inbound expressions of interest
- at least one or two contributors beyond the founder
- qualitative feedback like "this is exactly what I have been thinking"

## Risks

### 1. Concept drift

Risk:

The site drifts into generic indie maker or AI build content.

Response:

- keep the editorial lens explicit
- curate tightly
- foreground personal friction and usefulness

### 2. Overbuilding too early

Risk:

Too many community/product features obscure the core idea.

Response:

- start with editorial + submissions only
- add community features later only if the concept proves itself

### 3. Weak examples

Risk:

If the first content is vague or too polished, the concept will feel generic.

Response:

- lead with specific, idiosyncratic, clearly useful builds
- make the friction explicit in each example

### 4. Over-indexing on tooling

Risk:

The site becomes more about AI coding tools than about personal software.

Response:

- treat tools as enabling context
- keep the focus on the resulting software and the problem it solves

## Build Recommendation

Build `One Active User` as a curated editorial website first.

Do not start by building a community platform.

The first version should prove:

- the concept is legible
- the framing is compelling
- the content model works
- people want to see and share personal software through this lens

If that works, future versions can add:

- contributor pages
- newsletters
- recurring series
- collections
- interviews
- richer discovery features

Create the new project with this specification and build a v1 around:
